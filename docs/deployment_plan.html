📋 Stock-Report v2.19 部署建议书

2026年6月26日 · 首席策略官审计 · 待执行计划

🎯 结论先行

可以投入数据收集。 系统架构完整、CI/CD 就绪、模块耦合可控。核心风险不是架构，是数据管道稳定性。

建议策略：先运行 30 天 积累真实数据 再回来验证模型

1. 系统健康度总览

维度评分判定说明
架构设计8/10健康18层管线分层清晰，模块职责单一
CI/CD 基建8/10健康4个cron + CI Gate + Pages，云端免费运行
算法逻辑6/10待验证思路正确，缺乏真实历史数据回测
数据质量3/10薄弱管道脆弱、多处随机数、LS 常返回 0
可维护性7/10健康文档齐全（DEV + PRD + GOVERNANCE）
实盘可用性1/10不可用无券商API、无模拟交易、无真实成交
✅ 结论：系统可以投入「观察运行」阶段。 它的定位目前是「量化监测工具」，不是「自动化交易系统」。在这个定位下，架构是合格的。

2. 三阶段部署路线图

Phase I：数据积累期（Day 1-30）当前
目标：让 4 个 cron 稳定运行 30 天，积累真实数据
Phase II：模型验证期（Day 31-45）
目标：用 30 天真实数据跑完整回测 + 消融分析
Phase III：信号优化期（Day 46-60）
目标：根据回测结果调参、删减冗余模块、确认核心信号

3. Phase I 详细操作清单

3.1 立即执行（Day 0，今天）

🔧 修复 3 个数据管道问题

问题修复影响
LS 龙头评分永远 0 换用 ak.stock_board_cons_ths 获取成分股；失败时用 ETF 持仓数据兜底 P0 否则龙头模块完全无效
Dashboard 用随机数 Cost Basis 和 Breakout 改用缓存数据 + 默认 fallback 标注「无实时数据」 P0 否则仪表盘数据不可信
AKShare 限频 4 个 cron 之间间隔 > 30 分钟（已有），加 time.sleep(3) 在批量 API 调用间 P1 降低 API 超时率

3.2 每日自动运行（Day 1-30）

📅 4 个 cron 任务

时间任务产出
08:15早盘报告美股收盘 + 日韩开盘 + 商品 + 新闻 → morning_report.html
11:35午盘分析市场广度 + 板块 + 持仓 → midday_report.html
15:05收盘总结六大指数 + 资金流向 + 期货 → closing_report.html

📊 关键数据积累： 每个 cron 执行时会自动更新 history_cache/（板块/股票日线），30 天后将累积约 14,000 条真实记录（475板块 × 5指标 × 30天 ≈ 71,000 数据点）。

3.3 每周五检查清单

✅ 每周五 16:00 人工检查项

  1. 打开 dashboard.html → 检查是否所有 11 张卡片都有数据（无一串 ? 或 N/A）
  2. 打开 audit_bundle.json → 检查 rotation_signals 数量是否合理（牛市应有信号，熊市应为 0）
  3. 打开 self_check_YYYY-MM-DD.json → 所有检查项应为 PASS，非 PASS 需排查
  4. 检查 GitHub Actions 运行历史 → 30 天成功率应 > 90%）
  5. 打开 weekly_report.html → 确认数据覆盖本周 5 个交易日

4. 风险清单与缓解措施

风险概率影响缓解计划
AKShare API 大面积失效 中 高 已有 Sina 双源 fallback；失效时 pipeline 优雅降级（填「数据暂缺」）
GitHub Actions 超限 低 中 当前用量约 66min/月，免费额度 2000min，有 30 倍余量

ML 模型过拟合/无泛化 中 中 当前 ML 权重仅 20%（混合模式），不主导决策；Phase II 回测确认后逐步提升
模块耦合导致单点故障 中 中 每个模块有独立 try/except + fallback；Meta Score 失败时默认 HOLD

6. 你的"不要做"清单

⚠️ Phase I 期间明确禁止：
  1. 不要添加新引擎/新模块 — 当前 18 层已经足够复杂，再加就是过度工程
  2. 不要用 dashboard 的数字做交易决策 — 里面还有随机数，等 Phase I 数据管道修复后再信
  3. 不要手动调权重 — 让 Dynamic Weight Learning 自己收敛，人工调参 = 过拟合
  4. 不要频繁切换模型版本 — 选一个版本（当前 v2.19），让它跑完整 30 天再评估
  5. 不要过早优化 — API 偶尔超时不致命，别花一周修一个一天遇到两次的 bug

7. 总结

最终建议

✅ 系统可以投入运行。先修 3 个数据管道 bug，然后让它稳定跑 30 天。

30 天后回来看回测结果，再决定哪些模块该留、哪些该砍、哪些需要重写。

当前阶段：「观测 → 积累 → 验证」循环。不要急于做交易。
